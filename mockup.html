📝
drafts
In-progress form saves
sessionIdUUID — identifies this form session, stored in Application on submit
resumeTokenToken for email resume link (legacy — OTP auth is now primary)
emailFrom form data
stepLast active step (1–4)
status"draft" | "signing" | "submitted"
formDataMixed — full form snapshot
lastActiveAtUsed by abandoned cron
Lifecycle: Created when email entered on Step 1. Auto-saved every 3s while user fills form. Marked "submitted" on final submit (not deleted, kept for audit). Looked up by email after OTP auth so returning users resume their draft on any device. Abandoned cron alerts sales after 24h inactivity.
🔢
counters
Auto-increment sequences
name"application_counter" (only one doc)
valueNumber — increments atomically via $inc
Usage: On every form submission, Counter.findOneAndUpdate({ name: "application_counter" }, { $inc: { value: 1 } }) returns the next number → formatted as BC-2026-0001. Always exactly 1 document in this collection.
🔐
salesconfigs
Sales portal auth
key"sales_password"
passwordHashbcrypt hash
updatedAtDate
Usage: Sales portal login hashes the SALES_PORTAL_PASSWORD env var into this collection on first use. JWT token issued on success, valid for 8h.
🍪
customersessions
Customer OTP + session tokens
emailString – indexed, one doc per email
otpHashHMAC-SHA256(otp, email) – null after use
otpExpiresAtDate – 10 min from send
sessionTokenUUID – stored in httpOnly cookie
sessionExpiresAtDate – 30 days from verify
createdAtDate – TTL index: auto-delete after 32 days
Cookie: bytescare_session — httpOnly, secure (prod), sameSite=lax. On each page load, /api/auth/session validates the token and returns the customer's draft or application state.

POST /api/auth/send-otp
Body: { email }
→ generates 6-digit OTP
→ HMAC-SHA256 hash stored in DB
→ emails OTP (expires 10 min)
→ rate-limited: 1 per 60s/email
POST /api/auth/verify-otp
Body: { email, otp }
→ timing-safe compare
→ issues UUID session token
→ Sets httpOnly cookie (30 days)
→ clears OTP from DB
GET /api/auth/session
Reads bytescare_session cookie
→ looks up CustomerSession
→ finds Application by email
→ or Draft by email
→ returns full state for UI
